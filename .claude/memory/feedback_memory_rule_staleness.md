When citing a rule from memory, check whether the conditions that made it true still apply. Don't reflexively apply rules whose context has shifted.

**Why:** On 2026-04-24 the YPP suspension memory had two rules — "Do NOT delete content" and "Do NOT bulk-edit metadata during the 90-day wait." Both were written when an appeal was pending — the rationale was "preserve state during active review." With both appeals dead and no review pending, the rationale no longer applied, but I cited the rules reflexively when Thomas wanted to clean up the catalog. He pushed back twice ("wasn't that before we got denied?"); both times he was right.

**How to apply:**
- When recalling a rule, also recall *why* it exists (the **Why:** line on feedback/project memories is there for this).
- If the user is operating in a different context than when the rule was written, name the conflict explicitly and ask whether the rule still applies.
- If the rationale is clearly obsolete (e.g. the event the rule was guarding against has already happened or been resolved), retire the rule rather than re-applying it. Update the source memory file so it doesn't trip the next session.
- This isn't a license to ignore rules — it's a license to check them. If you can't articulate why a rule still applies, that's a signal to verify before citing.
